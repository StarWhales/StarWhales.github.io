—信息收集知识

状态码：

状态码 状态描述 含义与应用场景
200 OK 请求成功，服务器返回请求的资源，是最常见的成功状态码
201 Created 请求已成功处理并创建了新资源（常用于 POST 请求创建资源，如新建用户、新增文章）
301 Moved Permanently 资源已永久移动到新位置，客户端应使用新 URL 进行后续请求（如网站域名更换）
302 Found 资源临时移动到新位置，客户端应继续使用原 URL（如临时跳转、活动页面临时迁移）
304 Not Modified 资源未修改，客户端可使用本地缓存的版本（用于 HTTP 缓存机制，减少重复请求）
400 Bad Request 请求语法错误或参数无效（如参数格式错误、必填参数缺失），服务器无法理解
401 Unauthorized 请求需要身份验证（如登录态失效、未携带 Token），客户端未提供或验证失败
403 Forbidden 服务器拒绝请求，客户端已通过身份验证，但没有目标资源的访问权限
404 Not Found 请求的资源不存在或已被移除（如访问不存在的页面、调用已删除的接口）
405 Method Not Allowed 请求使用的方法不被服务器支持（如用 POST 访问仅支持 GET 的接口、用 PUT 访问仅支持 DELETE 的接口）
500 Internal Server Error 服务器内部发生错误（如代码逻辑异常、数据库连接失败），无法完成请求处理
502 Bad Gateway 网关或代理服务器从上游服务器（如后端业务服务器）收到无效响应
503 Service Unavailable 服务器暂时无法处理请求（通常是服务器维护、资源耗尽，一般为临时状态）

sqlmap参数

目标指定

  • -u “携带参数的 url”
  • -r req.txt（保存请求包的文本）
  • -m urls.txt（保存多个 url 的文本）

注入检测

  • -p “id” 指定参数

  • –dbms mysql 指定数据库类型

  • –technique=UBTES

    指定注入技术

    • U：union 查询
    • B：布尔盲注
      • –string “…” 表示正常情况下回显的字符
    • T：时间盲注
    • E：报错注入
    • S：堆叠查询

  • –level 1（1-5，数字越高 payload 越多）

  • –risk 1（1-3，数字越高 payload 越危险，可能有增删改的语句）

数据提取

  • –dbs 枚举所有数据库
  • –current-db 获取当前数据库名
  • -D “…” 指定数据库名
  • –tables 枚举指定数据库的表名
  • -T “…” 指定表名
  • –columns 枚举指定表的列名
  • -C “…,…” 指定字段名
  • –dump 提取指定列的数据
  • –dump-all 提取所有数据库数据

输出控制

  • –output-dir “…” 指定结果保存目录（默认在 output/ 下）
  • –threads 1 设置并发线程数（默认 1，最大 10，过高可能被 WAF 拦截）

绕过模板

  • –tamper “…” sqlmap -u "http://example.com/page?id=1" --tamper "space2comment,equaltolike,chardoubleencode,randomcase" --batch

ssrf

漏洞场景

SSRF漏洞一般存在于Web应用程序中，这些应用程序接受来自用户的输入，然后将其用于向其他服务

器发出请求。攻击者可以在输入中注入恶意的URL，从而使服务器发起未经授权的请求，以访问敏感

的内部资源。

  • 文件上传功能：Web应用程序通常允许用户上传文件，攻击者可以上传包含恶意URL的文件，以触发SSRF漏洞。
  • 图片处理功能：Web应用程序通常包含图片处理功能，攻击者可以在图片URL中注入恶意的URL，以触发SSRF漏洞。
  • URL重定向功能：Web应用程序可能包含URL重定向功能，攻击者可以在重定向URL中注入恶意的URL，以触发SSRF漏洞。
  • API调用：Web应用程序可能会使用API与其他服务进行交互，攻击者可以在API请求中注入恶意的URL，以触发SSRF漏洞。

Apache Shiro-550漏洞

在Apache shiro的框架中，执行身份验证时提供了一个记住密码的功能（RememberMe），如果用户登录时勾选了这个选项。用户的请求数据包中将会在cookie字段多出一段数据，这一段数据包含了用户的身份信息，且是经过加密的。加密的过程是：用户信息=>序列化=>AES加密（这一步需要用密钥key）=>base64编码=>添加到RememberMe Cookie字段。勾选记住密码之后，下次登录时，服务端会根据客户端请求包中的cookie值进行身份验证，无需登录即可访问。服务端进行对cookie进行验证的步骤就是：取出请求中rememberMe的cookie值 => Base64解码=>AES解密（用到密钥key）=>反序列化

因此攻击者只要找到AES加密的密钥，就可以构造一个恶意对象，对其进行序列化–>AES加密–>Base64编码，然后将其作为cookie的rememberMe字段发送，Shiro将rememberMe进行解密并且反序列化，最终造成反序列化漏洞

查看响应包头部是否有Set-Cookie: rememberMe=deleteMe（可以手动在请求包的Cookie处添加rememberMe=1），出现rememberMe=deleteMe字段仅仅能说明登录页面采用了shiro进行了身份验证而已，并非就说明100%存在漏洞

用shiro的java工具一键爆破密钥和rce

(1) Send with Cookie 开启Cookie，可以把包的Cookie存下来，生成请求的时候保留Cookie。

(2) Auto Request Sending 开启对API的请求，开启后可能会访问delete等可能危害服务器的接口，选择开启Auto Request Sending后，可以对子API进行自动化鉴权测试，快速发现API未授权访问漏洞。
