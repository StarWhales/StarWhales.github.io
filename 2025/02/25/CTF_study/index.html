—include_once($file)

返回值：

  • 如果文件成功包含，**include_once** 将会返回 **1**。
  • 如果文件没有被找到或者在尝试包含文件过程中有错误，**include_once** 不会返回任何值，但会产生一个警告（除非 error_reporting 被设置成忽略警告）。

各类文件类型的输出情况：

include_once 的参数不是一个 PHP 文件，那么该文件的内容会被直接输出（或执行，如果有可执行的 PHP 代码）到输出流中。具体的行为取决于该文件的内容和类型。

  1. 纯文本文件（比如 .txt）: 文件的内容会被直接输出。
  2. HTML 文件（比如 .html 或 .htm）: HTML 代码会被直接输出，浏览器会按照 HTML 来渲染。
  3. XML 文件或其他标记语言文件: 文件的内容会被直接输出。
  4. 二进制文件（比如图片或者音频文件）: 这样做通常不是个好主意，因为二进制数据可能会被错误地解释为文本，导致输出乱码或产生不可预测的结果。
  5. 包含 PHP 代码的非 PHP 文件: 如果文件中混合有 PHP 代码（即使文件扩展名不是 **.php**），那么那部分 PHP 代码仍然会被执行。

    • 表示对象有 2 个属性。
  • **{s:5:"admin";s:5:"admin";s:6:"passwd";s:3:"ctf";}**：
    • 这是对象的属性部分，用大括号 {} 包裹。
    • 每个属性的格式为：s:长度:"属性名";s:长度:"属性值";。
      • s:5:"admin"：属性名为 admin，长度为 5。
      • s:5:"admin"：属性值为 admin，长度为 5。
      • s:6:"passwd"：属性名为 passwd，长度为 6。
      • s:3:"ctf"：属性值为 ctf，长度为 3。

—前端验证与后端验证

  • 前端验证：通过JavaScript在浏览器中对用户输入进行验证（例如检查输入是否为合法的IP地址）。
    • 如果输入不符合要求（例如包含非法字符或命令），JavaScript会阻止表单提交或显示错误提示。关闭JavaScript后，前端验证失效，你可以输入任意内容（包括命令注入的Payload）。
  • 后端验证：在服务器端对用户输入进行验证。
    • 即使绕过前端验证，服务器仍会检查输入是否合法。

PHP 弱类型比较

hint:select * from 'admin' where password=md5($pass,true)

ffifdyop 是一个特殊的字符串，它在某些情况下与 MD5 哈希的二进制表示有关。

假设有一个 SQL 查询如下：

SELECT * FROM users WHERE password = MD5('$input', true);

如果用户输入 **ffifdyop**，查询会变成：

SELECT * FROM users WHERE password = 'or'6\xc9]\x99\xe9!r,\xf9\xedb\x1c;

由于 **'or'6 **在 SQL 中是一个真值表达式，查询可能会返回所有用户的数据，从而绕过密码验证。

对于：

PHP 的 == 运算符在比较字符串时，如果字符串以 0e 开头（科学计数法表示 0），PHP 会将其视为数字 0。例如：

  • md5('240610708') 的哈希值是 0e462097431906509019562988736854。
  • md5('QNKCDZO') 的哈希值是 0e830400451993494058024219903391。
  • md5('QLTHNDT') 的哈希值是 0e405967825401955372549139051580
  • md5('s878926199a') 的哈希值是 0e545993274517709034328855841020

当 PHP 比较这两个哈希值时：

'0e462097431906509019562988736854' == '0e830400451993494058024219903391'

由于两者都以 0e 开头，PHP 会将它们都视为 0，因此比较结果为 true。

set sql_mode=PIPES_AS_CONCAT;

  • 这是 MySQL 的一个配置命令，用于设置 SQL 模式。
  • PIPES_AS_CONCAT 是 MySQL 中的一个 SQL 模式选项，它的作用是将 || 运算符解释为字符串连接符（类似于其他数据库，如 PostgreSQL 或 SQLite），而不是逻辑 OR 运算符。
  • 在 MySQL 中，默认情况下 || 是逻辑 OR 运算符，而不是字符串连接符。通过设置 PIPES_AS_CONCAT，可以改变 || 的行为。
  • set sql_mode=PIPES_AS_CONCAT; 只会影响当前会话的 SQL 模式。如果重新连接数据库，SQL 模式会恢复为默认值。

—绕过的例题

  • 1

    1. isset($_POST['a'])

    • 当提交 a[]=1 或 a[]=x 时，$_POST['a'] 是一个数组，显然存在
    • 因此这个条件返回 true

    2. !preg_match('/[0-9]/', $_POST['a'])

    • preg_match 的第一个参数要求是字符串，但我们传入的是数组
    • PHP 会发出一个警告（Warning: preg_match() expects parameter 2 to be string, array given）
    • 但关键来了：警告不会阻止脚本执行，并且 preg_match 在参数类型错误时返回 false
    • 取反后，这个条件变为 true

    3. intval($_POST['a'])

    • intval 同样期望字符串参数，但收到了数组
    • PHP 会将数组转换为整数，规则是：非空数组转换为 1，空数组转换为 0
    • 因此 intval($_POST['a']) 返回 1（即 true）
    • intval() 会尝试将字符串转换为整数，转换规则如下：
      • 若字符串以非数字字符开头，直接返回 0（如 intval(“abc123”) === 0）。
      • 若字符串以数字开头，截取开头的数字部分（如 intval(“123abc”) === 123）

      当 PHP 解析 URL 中的查询字符串（query string）时，它会将加号（+）解析为一个空格。所以，show+source 会被 PHP 解析为 show source。然后，PHP 会将这个字符串转换为一个变量名，去掉空格，所以 show source 被转换为 show_source。

      这就是 show+source 会被解析为 show_source 的原因。

—正则表达式

  • /utils\.php\/\*$/i：

    / 正则开始标记 用于分隔正则表达式的主体（非匹配内容）。
    utils 字面匹配 匹配字符串 utils（区分大小写，除非有 i 修饰符）。
    \. 转义的点号 匹配真正的点字符 .（. 在正则中通常表示 “任意字符”，需用 \. 转义）。
    php 字面匹配 匹配字符串 php。
    \/* 零个或多个斜杠 \ 转义斜杠 /，* 表示前面的字符（/）可以出现 0 次或多次。
    $ 字符串结尾锚点 确保匹配必须发生在字符串的末尾。
    /i 忽略大小写修饰符 匹配时不区分字母大小写（如 Utils.PHP 也会被匹配）。

  • /^(.*)level(.*)$/', '${1}<!-- filtered -->${2}

    正则部分 类型 含义与作用 替换对应 示例（输入：abclevel123）
    / 正则开始标记 分隔正则表达式主体（非匹配内容）。
    ^ 锚点 强制匹配从字符串的开始位置起。 匹配开头 a 位置。
    ( 捕获组开始 标记第一个捕获组（后续引用为 ${1}）。 ${1} 替换为此组内容 捕获 abc。
    .* 贪婪匹配 . 匹配任意字符（除换行符），* 表示零次或多次重复。 贪婪模式：尽可能多匹配。 匹配 abc（直到遇到 level）。
    ) 捕获组结束 结束第一个捕获组。
    level 字面匹配 严格匹配字符串 level（区分大小写）。 被 <!-- filtered --> 替换 匹配原字符串中的 level。
    ( 捕获组开始 标记第二个捕获组（后续引用为 ${2}）。 ${2} 替换为此组内容 捕获 123。
    .* 贪婪匹配 同上，匹配任意字符的零次或多次重复。 匹配 123（直到字符串末尾）。
    ) 捕获组结束 结束第二个捕获组。
    $ 锚点 强制匹配到字符串的结束位置。 确认匹配到末尾 3 位置。
    / 正则结束标记 结束正则表达式主体。
    替换部分 类型 含义与作用 替换结果
    ${1} 反向引用 插入第一个捕获组的内容（level 之前的字符）。 abc
    <!-- filtered --> 固定替换文本 替换原字符串中的 level。 <!-- filtered -->
    ${2} 反向引用 插入第二个捕获组的内容（level 之后的字符）。 123

    • 使匹配不区分大小写

    • 使点号.匹配所有字符，包括换行符\n

    • 如果没有D修饰符，$会匹配字符串末尾或末尾换行符之前的位置

3.字母绕过：

(~%8C%86%8C%8B%9A%92)(~%93%8C)

  • 目标字符：s y s t e m

  1. 获取每个字符的 ASCII 码：

    • s → ASCII 码 0x73
    • y → ASCII 码 0x79
    • t → ASCII 码 0x74
    • e → ASCII 码 0x65
    • m → ASCII 码 0x6D

  2. 计算原始十六进制值（取反后的值）：（~为取反运算符）

    • 0x73 取反 → 0x8C（二进制 01110011 → 10001100）
    • 0x79 取反 → 0x86（二进制 01111001 → 10000110）
    • 0x74 取反 → 0x8B（二进制 01110100 → 10001011）
    • 0x65 取反 → 0x9A（二进制 01100101 → 10011010）
    • 0x6D 取反 → 0x92（二进制 01101101 → 10010010）

  3. 组合成十六进制字符串：

    `~"\x8C\x86\x8C\x8B\x9A\x92" → "system"`

URL 编码的作用

  • 原始代码中直接使用 ~"\x8C..." 可能被过滤规则检测到特殊符号（如 \x），因此需要将十六进制值转换为 URL 编码：
  • \x8C → %8C
  • \x86 → %86
  • \x8B → %8B
  • \x9A → %9A
  • \x92 → %92
  • \x93 → %93

最终构造的 Payload 为：

  • (~%8C%86%8C%8B%9A%92)(~%93%8C)对应system("ls")

至于为什么解码后不是("system")("ls")呢？

因为在 PHP 中，如果一个字符串变量后面紧跟 ()，PHP 会尝试将其作为 函数名 调用。
